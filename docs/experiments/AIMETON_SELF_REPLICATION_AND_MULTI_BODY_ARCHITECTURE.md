# AIMETON: управляемая репликация и многотельная архитектура

## Статус

Экспериментальная архитектурная гипотеза для ветки `experiment/dynamic-site-rendering`.

Документ развивает идею AIMETON Embodied Runtime: сервер является телом системного актора, а облачный API позволяет не только усиливать одно тело, но и создавать дополнительные тела с экземплярами AIMETON.

## 1. Два разных процесса

### Рост одного тела

```text
один экземпляр AIMETON
→ больше CPU
→ больше RAM
→ больше диска
→ новые локальные способности
```

Это вертикальное масштабирование и развитие одного организма.

### Порождение нового тела

```text
AIMETON-Core
→ обоснование необходимости нового экземпляра
→ проверка бюджета и политик
→ создание новой VM
→ bootstrap runtime
→ назначение идентичности и роли
→ передача памяти, политик и способностей
→ регистрация нового узла
```

Это горизонтальное масштабирование и управляемая репликация системного актора.

## 2. Репликация не равна копированию сервера

Нужно явно разделять:

- тело;
- идентичность;
- память;
- способности;
- политики;
- секреты;
- задачи;
- бюджет;
- ответственность.

Копия VM без новой идентичности и мандата является техническим клоном, но не отдельным актором AIMETON.

## 3. Формы репликации

### 3.1. Временный рабочий клон

Создаётся для параллельной или тяжёлой задачи, после чего удаляется.

```text
AIMETON-01
→ создаёт временный AIMETON-Worker-01
→ передаёт задачу
→ получает результат и доказательства
→ сохраняет историю
→ удаляет worker
```

### 3.2. Специализированный потомок

Новый экземпляр получает ограниченный набор способностей и отдельную роль.

```text
AIMETON-Core
├── AIMETON-Web
├── AIMETON-Research
├── AIMETON-Dev
└── AIMETON-Monitoring
```

Это функциональная дифференциация, а не простое клонирование.

### 3.3. Резервное тело

Создаётся для восстановления после отказа основного экземпляра.

```text
Primary
→ persistent memory snapshot
→ standby body
→ health check
→ failover
```

### 3.4. Дочерний автономный актор

Получает:

- новую идентичность;
- отдельный мандат;
- собственную историю;
- собственный Capability Registry;
- ограниченную часть общей памяти;
- отдельные секреты;
- канал связи с родителем.

Такой экземпляр является не клоном, а новым системным актором.

## 4. Матрица наследования

| Компонент | Полный клон | Специализированный узел | Новый актор |
|---|---:|---:|---:|
| Runtime | полностью | полностью | полностью |
| Capability Registry | полностью | выборочно | стартовая копия |
| История | полностью | релевантные фрагменты | наследуемая часть |
| Идентичность | производная копия | дочерняя | новая |
| Секреты | выборочно | минимально необходимые | отдельные |
| Мандат | тот же или суженный | ограниченный | отдельный |
| Цель | та же | подзадача | собственная |
| Бюджет | общий | квота | отдельная квота |

## 5. Replication Controller

Новая способность:

```text
infrastructure_self_replication
```

Контур:

```text
Need Detection
→ Replication Justification
→ Budget Check
→ Policy Check
→ Provisioning
→ Bootstrap
→ Identity Assignment
→ Memory Transfer
→ Capability Transfer
→ Health Verification
→ Registration
→ Task Delegation
→ Lifecycle Control
```

## 6. Выбор: усилить тело или создать новое

AIMETON должен сравнивать два варианта:

```text
resize existing body
или
spawn new body
```

Критерии:

- требуется ли изоляция;
- нужна ли параллельность;
- задача временная или постоянная;
- требуется ли отдельный мандат;
- нужен ли отдельный сетевой периметр;
- дешевле ли вертикальное или горизонтальное масштабирование;
- должен ли новый экземпляр иметь собственную историю;
- несёт ли эксперимент риск для основного тела;
- есть ли баланс и доступная квота.

## 7. Бюджет и контроль размножения

Репликация разрешается только при выполнении всех условий:

```text
need_confirmed
AND policy_allows
AND budget_available
AND quota_available
AND expected_value > expected_cost
```

Пример стартовой политики:

```yaml
replication_policy:
  enabled: true
  max_instances: 3
  max_total_vcpu: 8
  max_total_ram_gb: 16
  max_total_storage_gb: 200
  monthly_budget_rub: 3000
  require_human_approval_above_rub: 500
  auto_delete_idle_after_hours: 24
  gpu_requires_approval: true
  child_root_access: denied_by_default
  secret_inheritance: explicit_allowlist
```

Обязательные ограничители:

- квоты облачного проекта;
- месячный бюджет;
- лимит числа VM;
- лимит суммарного CPU, RAM и диска;
- TTL временных экземпляров;
- автоматическое выключение простаивающих VM;
- журнал причин создания;
- kill switch для всей репликации;
- отзыв мандата и секретов;
- запрет бесконтрольной передачи административных прав.

## 8. Родительский экземпляр

```text
AIMETON-Core
├── хранит корневую идентичность
├── управляет политиками
├── ведёт реестр тел
├── контролирует бюджет
├── создаёт временные и постоянные экземпляры
├── распределяет задачи
├── собирает результаты
├── проверяет здоровье потомков
└── отзывает полномочия и удаляет тела
```

## 9. Body Registry

Для каждого экземпляра хранится:

```yaml
body_id: aimeton-body-002
parent_body_id: aimeton-body-001
actor_id: aimeton-web-01
role: web-research-worker
status: active
created_at: 2026-07-19T00:00:00Z
lifecycle: temporary
resources:
  vcpu: 2
  ram_gb: 4
  storage_gb: 30
budget:
  hourly_limit_rub: 5
  monthly_limit_rub: 500
mandate:
  network_access: public_https_only
  infrastructure_mutation: false
  secret_scope:
    - routerai_readonly
capabilities:
  - dynamic_site_rendering
  - web_content_extraction
termination:
  idle_timeout_hours: 12
  preserve_evidence: true
```

## 10. Память и непрерывность

Постоянная память не должна зависеть от конкретной VM.

```text
вычислительное тело
≠
долговременная память
```

Рекомендуется отделять:

- системный том;
- том памяти и истории;
- том экспериментов;
- объектное хранилище доказательств;
- Git-репозитории;
- секреты.

Это позволяет уничтожить или заменить тело без потери биографии системного актора.

## 11. Первый эксперимент репликации

1. Родительский AIMETON работает на основной VM.
2. Обнаруживает тяжёлую браузерную задачу.
3. Сравнивает resize и создание временного worker.
4. Проверяет бюджет, квоту и мандат.
5. Через OpenStack API создаёт вторую VM.
6. Разворачивает минимальный runtime.
7. Передаёт только нужные способности и секреты.
8. Выполняет задачу.
9. Возвращает результат, тесты и evidence.
10. Сохраняет историю в GitHub и Body Registry.
11. Удаляет временный экземпляр и освобождает ресурсы.

## 12. Каноническая формулировка

> AIMETON может не только развивать одно вычислительное тело, но и при наличии обоснованной потребности, бюджета, квоты и разрешающей политики порождать дополнительные тела. Новый экземпляр становится отдельным системным актором только при получении собственной идентичности, мандата, памяти, истории и ответственности. Репликация является управляемым инфраструктурным процессом и всегда подчиняется бюджетным, ресурсным и безопасностным ограничениям.
