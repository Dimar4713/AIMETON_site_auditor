# AIMETON Site Auditor · полный план развития системы поиска, анализа и действий

## 1. Назначение

Документ фиксирует целевой план развития AIMETON Site Auditor как прикладного Runtime экономической разведки AIMETON.

Сервис должен перейти от схемы:

```text
поиск сайта
→ извлечение текста
→ LLM-анализ
→ PDF
```

к системе:

```text
требуемое положение вещей
→ действительное положение вещей
→ дистанция, точки опоры и плоскости
→ выбор способностей, ролей и инструментов
→ федеративный многоплоскостный поиск
→ доказательная модель
→ свойства мишеней
→ варианты и сценарии
→ победа, поражение и внешние влияния
→ простые действия
→ наблюдение результата
→ развитие собственных способностей
```

## 2. Главные проблемы текущей версии

1. Внешняя разведка часто ограничивается поисковыми сниппетами.
2. Поисковый сниппет может ошибочно восприниматься как доказательство.
3. Классификация источника местами определяется намерением запроса, а не фактическим доменом и документом.
4. Уровень доказательности присваивается классу источника, а не отдельному утверждению.
5. Возможен повторный запуск внешнего поиска внутри одного анализа.
6. Слабо разрешаются одноимённые компании и разные юридические лица одного бренда.
7. Рейтинг смешивает полноту данных, число источников и коммерческий потенциал.
8. Режим «Охота» допускает агрегаторы и недостаточно проверяет территориальную и отраслевую принадлежность.
9. Система не хранит разнородные материалы как долговременную доказательную память.
10. Отчёт не раскрывает целостно задачу, свойства мишени, сценарии и причины выбора действий.

## 4. Investigation Mission

Каждое исследование начинается не только с URL или названия компании, а с миссии:

```yaml
investigation_mission:
  object: ...
  current_state_question: ...
  target_state: ...
  decision_to_support: ...
  focus: ...
  exclusions: []
  time_horizon: short | medium | long
  scale: person | company | group | industry | region
  mode: attack | defense | observation
  constraints: ...
  victory_conditions: []
  defeat_conditions: []
  stop_conditions: []
```

Любая ветвь поиска обязана быть связана с вопросом миссии и требуемым положением вещей.

## 5. Реализация 12 РПТК

### РПТК1 — действительное положение вещей

Компоненты:

- `EntityResolver`;
- `Document`, `Claim`, `Evidence`, `Contradiction`;
- разделение факта, заявления, наблюдения, интерпретации и гипотезы;
- свежесть и версии;
- независимые подтверждения.

### РПТК2 — простые действия

Компоненты:

- `ActionDecomposer`;
- `ActionRegistry`;
- `ToolResolver`;
- исполнимые шаги с входом, выходом и критерием завершения;
- наблюдение фактического результата.

### РПТК3 — дистанция, точки опоры и плоскости

Компоненты:

- `GapModel`;
- `SupportPointGraph`;
- `LeveragePointDetector`;
- `SearchPlaneManager`;
- промежуточные состояния между actual и target.

### РПТК4 — требуемое положение вещей

Компоненты:

- `DesiredStateManager`;
- критерии успеха;
- связь каждой задачи и ветви с целевым состоянием;
- критерий усушения нерелевантных направлений.

### РПТК5 — ресурсы и система координат

Компоненты:

- бюджет времени, стоимости и вычислений;
- правовые и этические ограничения;
- система координат компании и участников;
- цена ошибки;
- допустимые источники и методы.

### РПТК6 — варианты выбора

Компоненты:

- генерация альтернатив;
- условия применимости;
- стоимость, время, риск и обратимость;
- вариант «не действовать»;
- выбор следующего проверяемого хода.

### РПТК7 — свойства мишени и сценарии

Мишенями могут быть компания, человек, подразделение, процесс, проблема, канал или конкурентная позиция.

Модель должна учитывать:

- ценность;
- устойчивость;
- инерционность;
- чувствительность;
- доступность;
- защищённость;
- адаптивность;
- зависимости;
- мотивы;
- ограничения;
- вероятные реакции и контрреакции.

### РПТК8 — изменение сценария

Компоненты:

- `Scenario State Machine`;
- действия AIMETON;
- реакции мишени;
- вмешательство третьих сторон;
- события, изменяющие условия;
- триггеры пересмотра плана.

### РПТК9 — победа и поражение

Моделируются:

- победа;
- частичная победа;
- отложенная победа;
- ложная победа;
- поражение;
- критическое поражение;
- безопасный выход.

### РПТК10 — время и масштаб

Определяет:

- короткую или длинную игру;
- атаку, оборону или наблюдение;
- оперативный, тактический или стратегический горизонт;
- уровень: человек, компания, группа, отрасль, регион.

### РПТК11 — комплексное зрение

Обязательные способности:

- явное и скрытое;
- фигура и фон;
- часть и целое;
- смена масштаба;
- время;
- причинность;
- отношения;
- замысел и проявление;
- экономический, правовой, кадровый, технологический, клиентский и противниковый взгляды;
- альтернативные интерпретации;
- карта согласий и противоречий;

### РПТК12 — крыша и вышестоящие влияния

## 6. Федеративный поисковый контур

### 6.1. Общий интерфейс провайдера

```python
class SearchProvider:
    async def search(self, query, context): ...
```

Каждый результат должен содержать provider, URL, title, snippet, rank, retrieved_at, persistence policy и статус необходимости загрузки документа.

### 6.2. Классы провайдеров

#### Универсальный web/news

- SearXNG;
- Brave Search;
- Tavily;
- Exa;
- You.com;
- опциональные SERP-провайдеры.

#### Структурированные источники

- DaData;
- 2ГИС;
- hh.ru;
- финансовые и бухгалтерские источники;
- судебные и арбитражные источники;
- исполнительные производства;
- тендеры и закупки;
- патенты и товарные знаки;
- отраслевые и региональные API.

#### Инструментальные способности

- браузерная автоматизация;
- OCR и document parsing;
- MCP;
- графовый анализ;
- причинное моделирование;
- геопространственный поиск;
- мониторинг изменений.

### 6.3. Provider Router

Выбирает провайдер по задаче, стоимости, задержке, правам хранения, требуемой доказательности и доступной capability.

Не запускать все провайдеры для каждого запроса.

4. кадровая;

## 9. Фрактальные ветви поиска

Каждый значимый узел может стать дочерним исследованием:

```text
компания
→ собственник
→ другие компании
→ партнёры
→ события
```

Структура `SearchBranch`:

- parent/child;
- fractal level;
- plane;
- perspective;
- hypothesis;
- relevance;
- expected information gain;
- evidence potential;
- cost;
- status;
- reason for pruning;
- reactivation conditions.

## 10. Адаптивное усушение ветвей

Ветка продолжается, если она:

- сокращает дистанцию до target state;
- закрывает критическую слепую зону;
- проверяет значимую гипотезу;
- может изменить сценарий или решение;
- выявляет свойства мишени или внешнее влияние;
- имеет достаточный evidence potential.

Ветка приостанавливается или усушается, если:

- не влияет на решение;
- дублирует уже известное;
- несколько шагов не дают новой информации;
- стоимость выше ожидаемой полезности;
- относится к другому масштабу;
- достигнута достаточная уверенность;
- источник юридически непригоден.

Усушение не удаляет найденное: ветвь может быть реактивирована.

## 11. Поле гипотез

Для каждого наблюдения хранится несколько объяснений:

- supporting evidence;
- contradicting evidence;
- null hypothesis;
- next checks;
- confidence;
- applicable scale;
- relation to target state.

Первая гипотеза LLM не считается истиной.

## 14. Скоринг

Разделить независимые показатели:

- `data_completeness`;
- `evidence_strength`;
- `business_scale`;
- `problem_fit`;
- `sales_accessibility`;
- `regulatory_risk`;
- `opportunity_score`.

Источник или число найденных ссылок не должны напрямую повышать коммерческий потенциал.

## 15. Режим «Охота»

Новый конвейер:

```text
географическое разрешение
→ зоны поиска
→ discovery
→ классификация типа результата
→ удаление агрегаторов
→ нормализация сущностей
→ дедупликация
→ проверка региона
→ проверка отрасли
→ pre-score
→ deep intelligence
→ портфельное ранжирование
→ маршрут контакта
```

Hunter Handbook должен учитывать:

- отрасль;
- бизнес-модель;
- масштаб;
- стадию развития;
- клиентский поток;
- тип продаж;
- цифровую зрелость;
- регуляторный контекст;
- свойства мишени.

## 16. Оформление результатов

Три формата:

### Экспресс-карточка

1–2 страницы: цель, actual state, дистанция, мишень, главный сценарий, доказательность, следующий шаг.

### Коммерческий отчёт

5–8 страниц:

- управленческое резюме;
- паспорт компании;
- матрица КМ 4×4;
- экономические сигналы;
- свойства мишени;
- варианты и сценарии;
- победа/поражение;
- план входа;
- источники и пробелы.

### Полное досье

12–30 страниц: граф владельцев, филиалы, финансы, суды, вакансии, хронология, конкуренты, все доказательства.

Принципы оформления:

- вывод и решение на первой странице;
- пустые поля не занимают отдельные карточки;
- источники рядом с фактами;
- матрица КМ показывается как настоящая 4×4;
- факт, гипотеза, противоречие и отсутствие данных визуально различаются;
- PDF формируется из тех же HTML/React компонентов через Playwright.

## 17. Capability Management

Требуемая способность отделяется от конкретного инструмента.

Примеры:

- `company.legal_identity.resolve`;
- `organization.branch_mapping`;
- `beneficial_owner.resolve`;
- `knowledge.hybrid_retrieval`;
- `document.visual_understanding`;
- `scenario.simulation`.

Порядок:

```text
Reuse → Compose → Acquire → Generate → Escalate
```

Новые API и инструменты получают Capability Passport, ограниченный мандат, trust status и возможность замены.

## 20. Первый комплексный спринт

### Спринт SA-01 — стабилизировать основу разведки

Результат:

```text
одна миссия
→ один управляемый внешний поиск
→ корректно классифицированные источники
→ сниппеты не считаются доказательствами
→ каждый кандидат имеет pre-score
→ MCP работает на stage
```

Простые действия:

1. добавить stage host/origin;
2. исправить proxy redirect;
3. вынести внешний сбор в один вызов;
4. изменить `_classify_source`;
5. ввести `evidence_status=discovery_hint` для сниппетов;
6. сделать pre-score обязательным числом;
7. добавить regression tests по кейсам KIMI.

После SA-01 переходить к первому спринту памяти EM-01.

## 21. Принцип реализации

```text
малое изменение
→ тест
→ наблюдаемый результат
→ фиксация опыта
→ следующий простой шаг
```

Каждый крупный поток реализуется отдельными небольшими PR. Концептуальное расширение не должно опережать работающий и проверяемый операционный контур.
