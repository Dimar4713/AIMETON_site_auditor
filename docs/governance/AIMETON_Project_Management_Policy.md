# AIMETON Site Auditor · политика и правила управления проектом

## Статус

Нормативный документ проекта.

Документ определяет, где и как фиксируются планы, движение работ, архитектурные основания, задачи, изменения, проверки и наблюдаемые результаты.

## 1. Цель политики

Обеспечить сквозную трассировку:

```text
назначение и архитектура AIMETON
→ направление развития
→ этап roadmap
→ Initiative / Epic
→ Issue
→ простые действия
→ ветка
→ Pull Request
→ тест
→ наблюдаемый результат
→ закрытие задачи
→ обновление плана и архитектуры
```

Главный принцип:

> Ни одна значимая работа не должна существовать отдельно от целевого состояния, архитектурного основания, исполнимой задачи и проверяемого результата.

## 2. Основная система управления

Оперативным центром управления проектом является **GitHub Projects**.

GitHub Projects используется для:

- текущего состояния работ;
- приоритетов;
- зависимостей;
- спринтов и итераций;
- дорожной карты;
- блокировок и рисков;
- связи Issues и Pull Requests;
- контроля завершения.

Отдельная внешняя система управления проектами на текущем этапе не вводится, чтобы не создавать второй источник истины.

## 3. Источники истины

| Объект | Источник истины |
|---|---|
| Каноническая архитектура AIMETON | `aimeton-architecture` |
| Исследовательские основания | `docs/research/` |
| План развития | `docs/roadmap/` |
| Политики и правила | `docs/governance/` |
| Архитектурные решения | ADR |
| Текущее состояние работ | GitHub Project |
| Единица работы | GitHub Issue |
| Реализованное изменение | Pull Request |
| Доказательство работоспособности | Tests / CI / validation evidence |
| Контрольное состояние продукта | Release |

GitHub Project не заменяет архитектуру и roadmap. Он показывает, какая часть замысла находится в реализации и каком состоянии она проявлена.

## 4. Иерархия работ

### 4.1. Initiative

Крупное направление развития системы.

Примеры:

- стабилизация Site Auditor;
- федеративная экономическая разведка;
- Evidence Memory и RAG;
- Раструб и 12 РПТК;
- Capability Management;
- отчётность и интерфейс;
- безопасность и эксплуатация.

### 4.2. Epic

Завершённый архитектурный или функциональный контур внутри Initiative.

Примеры:

- надёжный поисковый pipeline;
- Evidence и Entity Core;
- SearchProvider Layer;
- сохранение оригиналов;
- Investigation Mission;
- свойства мишеней и сценарии.

### 4.3. Issue

Основная управляемая единица работы.

Issue должна описывать один наблюдаемый результат, а не абстрактное намерение.

### 4.4. Sub-issue / checklist

Простые действия РПТК2, из которых состоит выполнение Issue.

## 5. Обязательные поля GitHub Project

Проект SHOULD содержать следующие поля:

| Поле | Назначение |
|---|---|
| `Status` | Backlog / Ready / In progress / Review / Validation / Done / Blocked |
| `Initiative` | крупное направление |
| `Epic` | архитектурный контур |
| `Stage` | P0–P9, M0–M10 или иной этап roadmap |
| `Priority` | P0 / P1 / P2 / P3 |
| `Type` | Research / ADR / Feature / Bug / Test / Infrastructure |
| `RPTK` | 1–12 / multiple / none |
| `Architecture link` | ссылка на нормативное основание |
| `Target state` | требуемое положение вещей |
| `Evidence of done` | Test / Demo / Metric / Review |
| `Iteration` | спринт или цикл |
| `Start date` | дата начала |
| `Target date` | целевая дата |
| `Effort` | 1 / 2 / 3 / 5 / 8 |
| `Risk` | Low / Medium / High |
| `Blocked by` | зависимость |
| `Capability` | требуемая способность |
| `Repository` | репозиторий выполнения |

## 6. Обязательные представления GitHub Project

### 6.1. Executive Roadmap

Отображает Initiatives и Epics на временной шкале.

### 6.2. Current Sprint

Kanban:

```text
Ready
→ In progress
→ Review
→ Validation
→ Done
```

### 6.3. Architecture Traceability

Таблица:

```text
Issue
| Architecture link
| RPTK
| Target state
| PR
| Test
| Observed result
```

### 6.4. Blockers and Risks

Фильтр задач со статусом `Blocked` или риском `High`.

### 6.5. Research Queue

Открытые вопросы, гипотезы и исследования, которые ещё не стали разработкой.

### 6.6. Capability Gaps

Задачи, для выполнения которых отсутствует инструмент, API, модель, метод или иной необходимый орган действия/восприятия.

### 6.7. Completed and Validated

Завершённые задачи, имеющие PR, тест и наблюдаемый результат.

## 7. Шаблон Issue

Каждая значимая Issue MUST содержать:

```markdown
## Требуемое положение вещей — РПТК4

Что должно стать истинным после выполнения задачи?

## Действительное положение вещей — РПТК1

Как система работает сейчас?

## Дистанция и препятствия — РПТК3

Что отделяет текущее состояние от требуемого?

## Архитектурное основание

Ссылка на архитектуру, roadmap, research note или ADR.

## Простые действия — РПТК2

- [ ] действие 1
- [ ] действие 2
- [ ] действие 3

## Критерии приёмки

- [ ] тест
- [ ] наблюдаемый результат
- [ ] отсутствие регрессии
- [ ] обновлена документация

## Риски и зависимости

...

## Объект изменения / мишень

Какой компонент, процесс или поведение изменяется?

## Evidence of Done

Ссылка на PR, тест, лог, скриншот, метрику или иной проверяемый результат.
```

## 8. Правила Pull Request

Каждый технический PR MUST быть связан с Issue.

При полном завершении используется:

```text
Closes #<issue-number>
```

PR MUST содержать:

```markdown
## Architecture

Ссылка на архитектурное основание.

## Target state

Какое требуемое состояние реализуется.

## Implemented

Что изменено.

## Tests

Какие проверки выполнены.

## Observed result

Какой наблюдаемый результат получен.

## Remaining gaps

Что осталось нерешённым.
```

Большие изменения выполняются через draft PR до завершения проверки.

## 9. Definition of Ready

Issue может перейти в `Ready`, если:

- определено требуемое состояние;
- зафиксировано текущее состояние;
- есть архитектурное основание;
- понятен наблюдаемый результат;
- указаны зависимости;
- задача разбита на простые действия;
- критерии приёмки проверяемы.

## 10. Definition of Done

Issue может перейти в `Done`, только если:

- изменение реализовано;
- PR слит либо документирован иной результат;
- тесты пройдены;
- наблюдаемый результат подтверждён;
- регрессии проверены;
- документация обновлена;
- остаточные ограничения зафиксированы;
- связанные задачи и roadmap актуализированы.

Закрытие Issue без Evidence of Done не допускается.

## 11. Жизненный цикл задачи

```text
Research / Observation
→ Backlog
→ Ready
→ In progress
→ Review
→ Validation
→ Done
```

Дополнительные состояния:

- `Blocked` — существует явная внешняя или внутренняя зависимость;
- `Cancelled` — задача более не соответствует цели;
- `Superseded` — заменена другой задачей или решением;
- `Deferred` — сознательно отложена.

Причина перехода в эти состояния MUST быть зафиксирована.

## 12. Трассировка Alpha → Omega

Для каждой значимой задачи необходимо обеспечить связь:

```text
Alpha / назначение
→ архитектурный принцип
→ roadmap
→ Issue
→ реализация
→ тест
→ Omega / наблюдаемый результат
```

Если результат не соответствует целевому состоянию, необходимо классифицировать расхождение:

1. дефект исполнения;
2. дефект модели;
3. дефект замысла.

Изменение верхнего архитектурного уровня требует отдельного Research/ADR, а не скрытой правки в техническом PR.

## 13. Правила работы с исследованиями

Исследовательские рассуждения сохраняются в `docs/research/`.

Research note SHOULD содержать:

- исходную проблему;
- наблюдения;
- альтернативы;
- аргументы за и против;
- принятое направление;
- оставшиеся открытые вопросы;
- последствия для roadmap.

Исследование не закрывается как Feature. После принятия решения оно порождает ADR, Epic или Issue.

## 14. Правила дорожной карты

Roadmap описывает целевой путь, но не заменяет Project.

При существенном изменении приоритетов необходимо:

1. обновить соответствующий roadmap;
2. изменить Initiative/Epic/Issue в GitHub Project;
3. зафиксировать причину изменения;
4. проверить влияние на зависимости;
5. сохранить прежнее решение в истории Git.

## 15. Правила малых изменений

Базовый цикл исполнения:

```text
малое изменение
→ тест
→ наблюдаемый результат
→ фиксация опыта
→ следующий простой шаг
```

Предпочтительны небольшие PR, каждый из которых:

- имеет одну основную цель;
- ограничивает область риска;
- легко проверяется;
- может быть откатан;
- оставляет систему в работоспособном состоянии.

## 16. Управление рисками и блокировками

Для риска `High` MUST быть указаны:

- причина;
- возможный ущерб;
- вероятность;
- способ снижения;
- условие остановки;
- ответственный контур решения.

Заблокированная задача не должна оставаться в `In progress`.

## 17. Capability Gaps

Если задачу нельзя выполнить доступными средствами, создаётся отдельная Issue класса `Capability Gap`.

Она должна описывать:

- требуемую способность;
- доступные способности;
- дефицит;
- ограничения;
- критерии успешного закрытия;
- порядок `Reuse → Compose → Acquire → Generate → Escalate`.

Новый инструмент не встраивается непосредственно в бизнес-логику без Capability Contract и адаптера.

## 18. Условия перехода к другой PM-системе

GitHub Projects используется до появления устойчивой потребности в:

- нескольких командах;
- бюджетах и ресурсном планировании;
- сложном портфельном управлении;
- внешних участниках без GitHub;
- полном self-hosted управленческом контуре;
- расширенной отчётности, недоступной в GitHub Projects.

При появлении таких требований рассматриваются Plane или OpenProject. До этого отдельная система не вводится.

## 19. Обязательный первый шаг внедрения политики

После принятия документа необходимо:

1. создать GitHub Project `AIMETON Development Control`;
2. создать обязательные поля;
3. создать представления;
4. перенести в Project этапы полного roadmap;
5. создать первые Initiatives и Epics;
6. оформить первый спринт `SA-01` как Issues;
7. связать будущие PR с соответствующими Issues.

## 20. Нормативный инвариант

> План считается управляемым только тогда, когда для каждого значимого направления видны его архитектурное основание, текущее состояние, следующий простой шаг, зависимость, доказательство выполнения и фактический результат.
