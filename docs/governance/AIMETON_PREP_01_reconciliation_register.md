# AIMETON Site Auditor · реестр предстартовой сверки PREP-01

## Статус

Нормативный документ предстартовой консолидации.

Документ фиксирует выявленные расхождения между ранними формулировками Site Auditor и актуальными каноническими уточнениями архитектуры AIMETON. При противоречии с более ранними формулировками документов PR №6 применяются решения настоящего реестра и актуальная архитектура `aimeton-architecture`.

## 1. Канонический статус РПТК11

### Выявленное расхождение

В ранних разделах полного плана встречается формулировка:

- `РПТК11 — комплексное зрение`;
- `Complex Vision`;
- каталог глаз как основное содержание РПТК11.

Эта формулировка отражает только познавательную часть и является неполной.

### Нормативное решение

Каноническое название:

> **РПТК11 · Собирательный образ**

РПТК11:

- собирает способы познания, точки зрения, масштабы и плоскости;
- собирает способности, роли, людей, агентов и инструменты;
- формирует комплексный образ действительного и требуемого положения вещей;
- действует комплексно интегрированным образом;
- включает частный профиль «смотрим всеми глазами», но не сводится к нему.

`Интегральная комплексная роль` может быть исполнительным носителем РПТК11, но не является его каноническим названием.

## 2. Три режима каждого РПТК

Каждый из 12 РПТК реализуется в трёх режимах:

1. `cognition` — способ познания, восприятия и формирования знания;
2. `action` — способ действия, созидания и преобразования;
3. `integrated` — связный цикл познания, действия, наблюдения результата и обратной связи.

Ни один РПТК не должен проектироваться как исключительно аналитический или исключительно исполнительный модуль.

## 3. Два столпа Раструба

Раструб опирается на два взаимодополняющих столпа:

- **знания** — способы и методы познания;
- **способности** — способы и методы действия.

Архитектура Site Auditor должна хранить и развивать не только сведения, доказательства и аналитические представления, но также конфигурации действий, роли, инструменты, исполнения и наблюдаемые исходы.

## 4. Evidence Memory — подсистема, а не вся система

### Выявленное расхождение

В ранней последовательности после SA-01 был указан прямой переход к EM-01.

### Нормативное решение

Обязательная последовательность:

```text
PREP-01
→ SA-01.1 baseline и минимальный CI
→ SA-01 стабилизация поискового и MCP-контура
→ Evidence и Entity contracts
→ Entity Resolution foundation
→ SearchProvider layer
→ первый внешний provider
→ первый structured provider
→ EM-01 сохранение оригиналов и provenance
→ новая аналитика и отчётность
→ дальнейшее развёртывание Раструба
```

Evidence Memory не заменяет:

- Investigation Mission;
- федеративный поиск;
- Entity Resolution;
- Evidence/Entity Graph;
- сценарный анализ;
- Capability Management;
- контур действий и наблюдения результата.

## 5. Статус полного плана

Документ `AIMETON_Site_Auditor_full_system_development_plan.md` остаётся главным планом системы.

Настоящий реестр:

- уточняет его терминологию;
- отменяет прямой переход `SA-01 → EM-01`;
- расширяет РПТК11 от зрения к собирательному восприятию и действию;
- вводит обязательный предстартовый контур baseline/CI;
- не изменяет общую цель и состав потоков P0–P9.

При следующем содержательном редактировании полного плана эти уточнения должны быть перенесены непосредственно в основной текст, после чего соответствующие пункты реестра могут быть отмечены как интегрированные.

## 6. Управление до создания GitHub Project

До физического создания проекта `AIMETON Development Control` действует временный нормативный режим:

- `CURRENT_STATUS.md` — быстрый указатель текущего шага;
- Issue #8 — активный PREP-01;
- Issue #7 — ожидающий Epic SA-01;
- Issues #9–#16 — подготовленные малые задачи SA-01;
- PR №6 — единый нормативный пакет;
- новые технические ветки запрещены до отдельного решения GO.

Отсутствие GitHub Project не блокирует merge нормативного пакета, если состояние и зависимости полностью отражены через Issues и `CURRENT_STATUS.md`.

## 8. Решение о техническом старте

Merge PR №6 завершает нормативную консолидацию, но сам по себе не запускает исправление runtime.

После merge текущим шагом становится:

> `SA-01.1 — Зафиксировать baseline и минимальный CI`

Изменение поискового и MCP-кода разрешается только после фиксации baseline и прохождения отдельной контрольной точки GO.
