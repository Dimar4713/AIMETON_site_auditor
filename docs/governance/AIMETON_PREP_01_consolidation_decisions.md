# AIMETON Site Auditor · PREP-01 · решения предстартовой консолидации

## Статус

Нормативный документ фазы PREP-01. До его завершения техническая реализация `SA-01` не начинается.

## 1. Цель

Привести архитектуру, исследования, roadmap, управление и главную ветку `main` к единому непротиворечивому состоянию, чтобы последующие технические ветки не расходились по разным трактовкам AIMETON.

## 2. Канонические уточнения, обязательные для Site Auditor

### 2.1. Два столпа Раструба

Раструб опирается на два взаимодополняющих столпа:

- **знания** — способы и методы познания, восприятия, различения, понимания и формирования образа действительности;
- **способности** — способы и методы действия, исполнения, созидания и преобразования действительности.

### 2.2. Три режима каждого РПТК

Каждый из двенадцати РПТК может использоваться:

1. в режиме `cognition` — как способ познания и восприятия;
2. в режиме `action` — как способ действия и преобразования;
3. в режиме `integrated` — как цикл познания, действия, наблюдения результата и обратной связи.

РПТК не делятся на исключительно познавательные и исключительно исполнительные.

### 2.3. РПТК11

Каноническое название:

> **РПТК11 · Собирательный образ**

РПТК11:

- собирает различные способы познания, точки зрения, масштабы и плоскости;
- собирает способности, роли, людей, агентов и инструменты;
- формирует не только комплексное видение, но и комплексно интегрированный способ действия;
- соединяет «смотрим всеми глазами» и «действуем всеми необходимыми способностями как единое целое».

Термин «комплексное зрение» допустим только как частный познавательный профиль РПТК11. Термин «интегральная комплексная роль» не является названием РПТК11 и может обозначать один из исполнительных носителей Собирательного образа.

## 3. Иерархия планов

Нормативный порядок документов:

```text
архитектура AIMETON
→ исследовательские основания Site Auditor
→ полный план развития Site Auditor
→ подсистемные планы
→ политика управления
→ CURRENT_STATUS
→ Issues
→ Pull Requests
→ Tests / Validation
```

Evidence Memory является подсистемой полного поисково-разведывательного контура и не заменяет:

- Investigation Mission;
- 12 РПТК;
- федеративный поиск;
- Entity Resolution;
- Evidence/Entity Graph;
- свойства мишеней;
- сценарии;
- действия и наблюдение результата.

## 4. Исправленная очерёдность запуска

До первой правки runtime:

```text
PREP-01
→ merge нормативного пакета в main
→ фиксация baseline main
→ минимальный CI
→ декомпозиция SA-01
→ SA-01: стабилизация
→ Evidence/Entity Core
→ SearchProvider Layer
→ первые провайдеры
→ EM-01: Raw Evidence Memory
→ новый отчёт
→ дальнейшее развёртывание Раструба
```

Прямой переход от `SA-01` к полному внедрению памяти запрещён: сначала должны появиться базовые Evidence/Entity contracts и управляемый Provider Layer. Допускается только раннее минимальное сохранение raw-артефактов за feature flag, если оно не меняет архитектурную очередность и не связывает бизнес-логику с конкретным RAG.

## 5. Правило главной ветки

- `main` содержит только согласованное и проверенное состояние.
- Новая техническая ветка создаётся только от зафиксированного актуального `main`.
- Нельзя продолжать разработку от старой экспериментальной ветки после изменения нормативного пакета.
- Перед merge проверяются документация, тесты, ссылки и отсутствие противоречий.
- После merge фиксируется стартовый SHA.
- `CURRENT_STATUS.md` обновляется в том же переходе состояния.

## 6. Baseline до разработки

До изменения runtime необходимо зафиксировать:

- commit SHA;
- версию приложения;
- зависимости;
- результат `pytest -q`;
- `/api/health`;
- контрольные ответы `/api/analyze`, `/api/hunt`, `/api/company-intelligence`;
- состояние `/mcp`;
- текущую конфигурацию доступных providers без раскрытия секретов;
- известные дефекты и ограничения;
- контрольные тестовые кейсы KIMI.

## 7. Минимальный CI

Обязательный первый CI-контур:

```text
pull_request / push
→ установка Python
→ установка зависимостей
→ pytest -q
→ проверка импорта app
→ минимальный smoke test API
```

CI должен появиться до первой серии технических исправлений, чтобы регрессия отличалась от исходного состояния.

## 8. Декомпозиция SA-01

`SA-01` остаётся эпиком стабилизации и раскладывается на независимые малые Issues:

1. `SA-01.1` — baseline и CI;
2. `SA-01.2` — MCP stage host/origin и redirect;
3. `SA-01.3` — устранение двойного внешнего поиска;
4. `SA-01.4` — исправление source/query classification;
5. `SA-01.5` — разделение discovery hint, source candidate и evidence;
6. `SA-01.6` — обязательный pre-score;
7. `SA-01.7` — регрессионный пакет KIMI;
8. `SA-01.8` — интеграционная проверка и фиксация результата.

Каждый технический PR закрывает одну малую Issue либо явно обоснованную тесно связанную группу.

## 9. Условия готовности PREP-01

PREP-01 завершён, когда:

- нормативная терминология согласована с актуальной архитектурой AIMETON;
- РПТК11 в документах называется «Собирательный образ»;
- для всех РПТК зафиксированы режимы cognition/action/integrated;
- устранены противоречия в очередности полного плана и памяти;
- созданы Issue/PR templates;
- SA-01 декомпозирован;
- определены baseline и CI;
- `CURRENT_STATUS.md` указывает правильный активный шаг;
- PR №6 переведён в Ready и объединён;
- зафиксирован SHA подготовленного `main`.

## 10. Правило допуска к старту

Техническая реализация начинается только после отдельного решения `GO`, основанного на выполненных критериях PREP-01. Сам факт наличия roadmap не является разрешением изменять runtime.
