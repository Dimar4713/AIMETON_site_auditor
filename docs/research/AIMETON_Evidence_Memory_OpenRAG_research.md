# AIMETON Site Auditor · Evidence Memory и первый RAG-контур

## Статус

Исследовательская архитектурная записка. Фиксирует решения, принятые после анализа тестирования KIMI, уточнённой архитектуры AIMETON и обсуждения фрактального многоплоскостного поиска через 12 РПТК.

## 1. Проблема

AIMETON Site Auditor в ходе разведки будет собирать разнородные материалы:

- HTML-страницы и снимки сайтов;
- PDF, DOCX, XLSX, CSV, JSON;
- поисковые результаты и ответы API;
- скриншоты, схемы, графики, таблицы и изображения;
- сведения о компаниях, людях, владельцах, связях и событиях;
- наблюдения разных когнитивных «глаз» РПТК11;
- гипотезы, противоречия, свойства мишеней и сценарии;
- промежуточные и итоговые отчёты.

Без самостоятельного слоя памяти значительная часть материалов будет теряться после формирования отчёта, повторно загружаться и анализироваться, а происхождение выводов станет трудно воспроизводимым.

## 2. Принятое решение

Первый практический контур памяти:

```text
AIMETON Evidence Memory v0.1
=
MinIO / S3-compatible object storage
+ PostgreSQL metadata and provenance
+ OpenRAG / OpenSearch hybrid retrieval
+ Docling document processing
+ Evidence Ingestion Gateway
+ REST / MCP access
```

OpenRAG рассматривается как первая заменяемая реализация способности `knowledge.hybrid_retrieval`, а не как единственное хранилище и не как целевая онтология уровня Palantir.

## 3. Архитектурный принцип

```text
Источник
→ неизменяемый оригинал
→ метаданные и происхождение
→ нормализованное представление
→ поисковый индекс
→ извлечённые утверждения
→ аналитические проекции
→ отчёт и действие
```

Любой поисковый индекс должен быть перестраиваем из оригиналов и метаданных.

Три инварианта:

1. Ничего значимого не теряется.
2. Для каждого вывода известен первоисточник.
3. RAG и модели можно заменить без утраты корпуса данных.

## 4. Почему одного RAG недостаточно

Обычная схема `документ → чанки → embeddings` не сохраняет полноценно:

- графическую структуру;
- точное расположение таблиц и диаграмм;
- версии страниц во времени;
- различие первоисточника, факта, гипотезы и рекомендации;
- связи сущностей;
- историю поисковых ветвей;
- provenance каждого когнитивного взгляда.

Поэтому OpenRAG является поисково-диалоговым слоем над Evidence Memory.

## 5. Слои памяти

### 5.1. Raw Object Store

Хранит оригинальные байты и снимки:

- HTML;
- PDF и офисные документы;
- JSON-ответы API;
- изображения страниц;
- скриншоты;
- экспортированные таблицы;
- архивные версии.

Каждый объект получает SHA-256 и content-addressed identifier.

### 5.2. Metadata and Provenance Store

PostgreSQL хранит:

- investigation_id, mission_id;
- source_id, document_id;
- URL и provider;
- запрос и поисковую ветвь;
- дату публикации, получения и версию;
- класс источника и evidence status;
- entity links;
- SearchPlane, Perspective и РПТК;
- retention/persistence policy;
- путь к оригиналу и индексам.

### 5.3. OpenRAG / OpenSearch

Используется для:

- Docling parsing;
- полнотекстового поиска;
- векторного и гибридного поиска;
- reranking;
- вопросно-ответного доступа;
- REST и MCP вызовов.

### 5.4. Evidence and Analysis Layer

Хранит отдельно:

- извлечённые наблюдения;
- проверенные claims;
- противоречия;
- гипотезы;
- свойства мишеней;
- сценарии;
- итоговые решения.

## 6. Разделение пространств знаний

Минимально требуются четыре логические коллекции:

1. `raw_evidence` — первичные и нормализованные источники;
2. `verified_claims` — подтверждённые утверждения;
3. `analytical_views` — гипотезы, РПТК-проекции и сценарии;
4. `investigation_outputs` — отчёты, решения и обратная связь.

Запрещено без явных меток смешивать первичный источник, интерпретацию LLM и итоговую рекомендацию.

## 7. Графические материалы

Для страниц PDF, изображений и диаграмм сохраняются четыре представления:

```text
original file
+ rendered page image
+ extracted text/structure
+ machine visual description
```

В OpenRAG индексируются текст, таблицы, подписи, OCR и ссылки на оригинальную страницу. Само изображение хранится в объектном хранилище. Полноценный Visual RAG добавляется позднее отдельной capability.

## 8. Фрактальная организация

Каждая значимая сущность формирует собственный контекст памяти:

```text
Компания
├── источники
├── факты
├── люди
├── филиалы
├── продукты
├── события
├── гипотезы
├── сценарии
└── исследования
```

Любой узел может раскрываться в дочернее исследование, сохраняя связь с родительской миссией.

## 9. Связь с 12 РПТК

Evidence Memory должна сохранять не только сведения об объекте, но и происхождение взгляда:

- РПТК1 — фактическое положение вещей;
- РПТК3 — дистанция, точки опоры и плоскости;
- РПТК4 — требуемое положение вещей;
- РПТК5/10 — ресурсы, система координат, время и масштаб;
- РПТК6 — варианты;
- РПТК7 — свойства мишеней и сценарии;
- РПТК8 — изменения сценариев;
- РПТК9 — победа и поражение;
- РПТК11 — комплексный синтез разных глаз;
- РПТК12 — внешние влияния и вышестоящие контуры.

Каждая аналитическая запись должна иметь provenance: кто/что её сформировало, на каких данных и в рамках какой миссии.

## 10. Связь с Capability Management

OpenRAG регистрируется как заменяемая capability implementation.

```yaml
capability:
  id: knowledge.hybrid_retrieval
  implementation: openrag
  status: verified_experimental
  replaceable: true
```

При обнаружении дефицита применяются правила:

```text
Reuse → Compose → Acquire → Generate → Escalate
```

Новые поисковые API, visual retrieval, graph reasoning и document parsers подключаются через Capability Registry, а не встраиваются напрямую в бизнес-логику.

## 11. Путь развития

```text
Document RAG
→ Evidence RAG
→ Entity-centric RAG
→ Graph RAG
→ Temporal and causal graph
→ Operational ontology
→ AIMETON Economic Intelligence Runtime
```

Целевая система не копирует Palantir сразу, но сохраняет данные и происхождение так, чтобы переход к объектной, графовой и операционной онтологии не требовал уничтожения первого решения.

## 12. Решение для MVP

Принято начать с:

- MinIO;
- PostgreSQL;
- OpenRAG/OpenSearch;
- Docling;
- Evidence Ingestion Gateway;
- раздельных коллекций;
- content hashing и versioning;
- ссылок из отчёта на первоисточники.

Первый критерий успеха: материалы одного исследования можно повторно найти, проверить и использовать в следующем исследовании без повторного полного сбора.
